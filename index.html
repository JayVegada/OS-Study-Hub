Semaphore is a synchronization primitive an integer variable accessed only through two atomic operations used to control access to shared resources and prevent race conditions.

Operations:

  • wait() / P(): Decrements the value. If value becomes negative, the process blocks.
  • signal() / V(): Increments the value. Awakens one waiting process if any.

Types:
Binary Semaphore: Value 0 or 1 used for mutual exclusion.
Counting Semaphore: Any integer used for multiple resource instances.

Dining Philosopher Problem

5 philosophers sit around a circular table. Each needs both left and right forks to eat. Objectives: avoid deadlock and starvation.

Basic Semaphore Solution:

semaphore fork[5] = {1,1,1,1,1};
do {
  think();
  wait(fork[i]);          // Pick left fork
  wait(fork[(i+1)%5]);    // Pick right fork
  eat();
  signal(fork[i]);        // Release left fork
  signal(fork[(i+1)%5]); // Release right fork
} while(TRUE);

Deadlock Issue: If all philosophers pick their left fork simultaneously, circular deadlock occurs.

Modified Solution (allow only 4 at table):

semaphore room = 4;
semaphore fork[5] = {1,1,1,1,1};
do {
  think();
  wait(room);
  wait(fork[i]); wait(fork[(i+1)%5]);
  eat();
  signal(fork[i]); signal(fork[(i+1)%5]);
  signal(room);
} while(TRUE);

ASCII Diagram:

      P0
   f0    f4
P4          P1
   f3    f1
      P3--f2--P2

Conclusion: The room semaphore (=4) guarantees at least one philosopher always gets both forks, preventing deadlock.

Banker's Algorithm is a deadlock avoidance algorithm that checks whether granting a resource request keeps the system in a safe state. Named after a banker who only grants loans when enough resources remain for all clients.

Key Terms:
Available: Currently free resources.
Max: Maximum resources a process may ever need.
Allocation: Resources currently given to a process.
Need = Max - Allocation: Still required resources.

Safe State: A state where a safe execution sequence exists so every process can finish.

Algorithm Steps:

  1. Compute Need matrix: Need[i] = Max[i] - Allocation[i]
  2. Find process P where Need[P] <= Available
  3. Simulate P completing: Available += Allocation[P]
  4. Repeat until all finish (SAFE) or no P found (UNSAFE)

Example (5 Processes, 3 Resources A, B, C):

Available = (3,3,2). Safe Sequence: P1 -> P3 -> P4 -> P0 -> P2

Step 1: P1 Need(1,2,2) <= (3,3,2). Execute. Available = (5,3,2).
Step 2: P3 Need(0,1,1) <= (5,3,2). Execute. Available = (7,4,3).
Step 3: P4 Need(4,3,1) <= (7,4,3). Execute. Available = (7,4,5).
Step 4: P0 Need(7,4,3) <= (7,4,5). Execute. Available = (7,5,5).
Step 5: P2 Need(6,0,0) <= (7,5,5). Execute. SAFE.

Conclusion: Banker's Algorithm prevents deadlock by ensuring resource requests are only granted when a safe sequence of execution exists.

Process is a program in execution an active entity managed by the OS. A program becomes a process when loaded into memory and executed by the CPU.

Components of a Process: Program Code, Data, Stack, Heap, Program Counter, CPU Registers, PCB.

Process States:

  1. New: Process is being created; OS allocates resources.
  2. Ready: Process loaded in memory, waiting for CPU.
  3. Running: CPU is actively executing process instructions.
  4. Waiting/Blocked: Process waiting for I/O or an event.
  5. Terminated: Process has finished; OS reclaims memory.

Disk Scheduling Algorithms decide the order to service disk I/O requests to minimize seek time and improve throughput.

Example Setup: Request Queue: 98, 183, 37, 122, 14, 124, 65, 67 | Initial Head: 53

1. FCFS (First Come First Serve)
Requests serviced in arrival order.

53 -> 98 -> 183 -> 37 -> 122 -> 14 -> 124 -> 65 -> 67

Total seek = |98-53|+|183-98|+|37-183|+... = 640 cylinders. Simple but high seek time.

2. SSTF (Shortest Seek Time First)
Service the nearest request to current head.

53 -> 65 -> 67 -> 37 -> 14 -> 98 -> 122 -> 124 -> 183

Total seek = 236 cylinders. Better performance. May cause starvation for far requests.

3. SCAN (Elevator Algorithm)
Arm moves in one direction servicing all requests, then reverses.

53 -> 65 -> 67 -> 98 -> 122 -> 124 -> 183 -> 199 -> 37 -> 14

|----[====================================]--------->|

|<--[===============]                                |
 14  37

4. C-SCAN (Circular SCAN)
Arm moves only one direction. After reaching end, jumps to beginning without servicing.

53 -> 65 -> 67 -> 98 -> 122 -> 124 -> 183 -> 199 (jump to 0) -> 14 -> 37

Disk scheduling is like an elevator deciding which floors to stop at. FCFS stops at every floor as requested. SSTF always goes to the nearest floor first. SCAN goes all the way up then all the way down. C-SCAN goes only up, then teleports to floor 0 and goes up again.

Key Points:
  • FCFS: simple, fair, high seek

  • C-SCAN: one direction + jump, uniform wait times
  • Seek time = head movement distance

RAID (Redundant Array of Independent Disks) combines multiple physical disks into one logical unit to improve performance, reliability, and fault tolerance using striping, mirroring, or parity.

RAID Level 0 Striping

Data is divided into blocks distributed across multiple disks simultaneously. No redundancy.

Disk 1:  A1  B1  C1
Disk 2:  A2  B2  C2
  • Speed: Maximum read/write (parallel access)
  • Fault Tolerance: None one disk fails = total loss
  • Storage Efficiency: 100%
  • Use: Gaming, video editing, high-performance apps

RAID Level 1 Mirroring

Identical data written to two or more disks simultaneously.

Data --> Disk 1 (primary)
Data --> Disk 2 (mirror)
  • Speed: Good reads, normal writes
  • Fault Tolerance: Excellent survives one disk failure
  • Storage Efficiency: 50%
  • Use: Banking, critical databases, servers

RAID Level 5 Striping with Distributed Parity

Data AND parity blocks distributed across all disks (min. 3 disks). Parity enables recovery.

Disk 1:  A1   B1   P3
Disk 2:  A2   P2   C1
Disk 3:  P1   B2   C2
  • Speed: High reads, moderate writes (parity overhead)
  • Fault Tolerance: Survives one disk failure
  • Storage Efficiency: (N-1)/N
  • Use: File servers, enterprise storage, web servers

💡 ELI10 (Plain Language)

RAID is like teamwork for hard drives. RAID 0: split your data across 2 drives super fast, but if one breaks, everything is gone. RAID 1: copy everything to 2 drives safe but wastes half your space. RAID 5: spread data AND a recovery code across 3+ drives if one fails, the code rebuilds it.

Key Points:
  • RAID 0: striping, no redundancy, 100% capacity, fastest
  • RAID 1: mirroring, full copy, 50% capacity, safest
  • RAID 5: distributed parity, (N-1)/N capacity, balanced
  • Parity = XOR of data blocks for recovery
  • Min disks: RAID 0=2, RAID 1=2, RAID 5=3
  • RAID 5 can survive exactly 1 disk failure
🧠 Memory Trick:

0=Zero protection. 1=One mirror. 5=Five-star balanced. Speed: 0>5>1. Safety: 1=5>0.

Deadlock is a situation where a set of processes are permanently blocked, each waiting for a resource held by another process in the set.

Coffman Conditions: Deadlock occurs if and only if ALL four conditions hold simultaneously.

1. Mutual Exclusion
At least one resource must be non-shareable only one process can use it at a time.
Example: Printer if P1 uses it, P2 must wait.

2. Hold and Wait
A process holding at least one resource is waiting to acquire additional resources held by other processes.
Example: P1 holds Resource A while waiting for Resource B held by P2.

3. No Preemption
Resources cannot be forcibly taken from a process only released voluntarily after completion.
Example: OS cannot steal a printer from P1 mid-print.

4. Circular Wait
A circular chain of processes exists: P1 waits for P2, P2 waits for P3, P3 waits for P1.

P1 --needs--> R2     R1 <--holds-- P1
P2 --needs--> R1     R2 <--holds-- P2

ConditionPrevention Technique
Mutual ExclusionMake resources shareable where possible
Hold and WaitRequest all resources before starting
No PreemptionAllow OS to preempt resources
Circular WaitImpose total resource ordering

Conclusion: Removing any one Coffman condition is sufficient to prevent deadlock. Circular wait is most commonly targeted via resource ordering.

Peterson's Solution is a classic software-based algorithm for achieving mutual exclusion between two processes without special hardware support.

Shared Variables:

boolean flag[2];  // flag[i]=true means Pi wants to enter CS
int turn;         // turn=i means Pi's turn to enter

Algorithm for Process P0:

do {
    flag[0] = true;
    turn = 1;
    while (flag[1] && turn == 1);  // busy wait
    /* Critical Section */
    flag[0] = false;
    /* Remainder Section */
} while (true);

Algorithm for Process P1:

do {
    flag[1] = true;
    turn = 0;
    while (flag[0] && turn == 0);  // busy wait
    /* Critical Section */
    flag[1] = false;
    /* Remainder Section */
} while (true);

Proof of Correctness (3 Properties):

1. Mutual Exclusion:
Suppose both P0 and P1 are in CS simultaneously. P0 in CS means: flag[1]=false OR turn==0. P1 in CS means: flag[0]=false OR turn==1. But turn can only be 0 OR 1 not both simultaneously. Contradiction. Hence only one process can be in CS at a time. PROVED.

2. Progress:
If no process is in CS and one wants to enter, it sets flag=true and gives turn to the other. If the other is not interested (flag=false), the waiting process enters immediately. No process outside CS blocks entry. PROVED.

3. Bounded Waiting:
A waiting process P0 gives turn=1 to P1. P1 executes its CS once, then sets flag[1]=false. P0 can then enter. P0 waits at most ONE turn of P1. No starvation. PROVED.

Limitations: Works for exactly 2 processes only. Busy waiting wastes CPU cycles. Not suitable for modern multiprocessors without memory barriers.

Conclusion: Peterson's Solution elegantly achieves mutual exclusion, progress, and bounded waiting for two processes using only shared memory variables.

TLB (Translation Lookaside Buffer) is a small, high-speed associative cache memory that stores recently used page table entries to speed up logical-to-physical address translation in paging systems.

Why TLB is Needed:
Without TLB: 2 memory accesses per instruction (1 for page table + 1 for data). TLB reduces this to nearly 1 memory access for frequent pages.

TLB Working:

CPU generates Logical Address (Page#, Offset)
        |
        v
    +-------+
    |  TLB  |  <-- Searched first (associative, parallel)
    +-------+
    /           HIT          MISS
   |             |
Frame#       Page Table in RAM
   |             |
   +-------+----+
           |
       Physical Address = Frame# + Offset
           |
       Main Memory Access

TLB Hit: Frame# found in TLB. Time = TLB lookup time (t) + Memory access (m).

TLB Miss: Not in TLB. Time = TLB time (t) + Page table access (m) + Memory access (m) = t + 2m.

Effective Access Time (EAT) Formula:

EAT = h*(t+m) + (1-h)*(t+2m)

Where: h = hit ratio, t = TLB lookup time, m = memory access time

Numerical Example:
Given: m=100ns, t=20ns, h=0.8 (80%)

EAT = 0.8*(20+100) + 0.2*(20+200)
= 0.8*(120) + 0.2*(220)
= 96 + 44
= 140 ns

Conclusion: TLB dramatically reduces effective memory access time. With 80% hit rate, EAT = 140ns vs 220ns without TLB a 36% improvement.

  item buffer[N];
  int in = 0, out = 0, count = 0;
  condition full, empty;

  procedure insert(item x) {
    if (count == N)
      wait(full);         // Buffer full, producer waits
    buffer[in] = x;
    in = (in + 1) % N;
    count++;
    signal(empty);        // Wake up sleeping consumer
  }

  procedure remove(item x) {
    if (count == 0)
      wait(empty);        // Buffer empty, consumer waits
    x = buffer[out];
    out = (out + 1) % N;
    count--;
    signal(full);         // Wake up sleeping producer
  }
}

Working:

  • Producer: If buffer full -> wait(full). Else insert item, signal(empty) to wake consumer.
  • Consumer: If buffer empty -> wait(empty). Else remove item, signal(full) to wake producer.

Advantages over Semaphores: Automatic mutex. Easier to reason about. No accidental misuse of wait/signal.

Conclusion: Monitor provides structured synchronization; Producer-Consumer is solved cleanly without explicit mutex management using condition variables.

Page Fault occurs when a process tries to access a page that is NOT currently in physical memory (RAM). The page exists in secondary storage (disk) and must be loaded.

Causes: Page not yet loaded. Page swapped out to disk. Invalid memory reference.

Page Fault Handling Steps:

  1. Trap: Hardware detects missing page, generates interrupt (trap) to OS.
  2. Validity Check: OS checks if the reference is valid. If invalid, terminate process.
  3. Locate Page: OS finds the required page on the disk (backing store).
  4. Free Frame: Find a free frame in RAM. If none, use page replacement algorithm (FIFO/LRU/Optimal).
  5. Load Page: Transfer page from disk to free frame in RAM.
  6. Update Page Table: Mark the page as present, record frame number.
  7. Restart: Resume the interrupted instruction.

Swapping is a memory management technique where an entire process is temporarily moved from RAM to secondary storage (disk) and later brought back for execution.

Terminology:

  • Swap Out (Roll Out): Moving process from RAM to disk.
  • Swap In (Roll In): Bringing process back from disk to RAM.
  • Swap Space: Dedicated area on disk for swapped processes.

Swapping Diagram:

+------------------+     Swap Out      +------------------+
|   Main Memory    | ----------------> |   Disk           |
|   (RAM)          | <---------------- |   (Swap Space)   |
+------------------+     Swap In       +------------------+

Working:

  1. Process P1, P2 are in RAM. P3 needs to execute but RAM is full.
  2. OS selects a process (e.g., P1) that is not actively using CPU.
  3. P1 is swapped out to disk (swap space).
  4. P3 is loaded into the freed RAM space and executes.
  5. Later, P1 can be swapped back in when P3 blocks or finishes.

Advantages: Supports multiprogramming. Better CPU utilization. Allows more processes than RAM can hold simultaneously.

Disadvantages: Disk access is very slow (milliseconds vs nanoseconds). High swap traffic causes thrashing. Context switch time increases.

Swapping vs Paging: Swapping moves ENTIRE process. Paging moves individual pages (more granular, preferred in modern OS).

Conclusion: Swapping extends effective memory by using disk as overflow storage, enabling multiprogramming at the cost of slower process resumption.

Keywords: Swapping • Swap Out • Swap In • Swap Space • RAM • Disk • Thrashing • Multiprogramming • Roll In Roll Out

Critical Section is a segment of code that accesses shared resources (variables, files, devices) and must not be executed by more than one process simultaneously.

  2. Progress: If no process is in CS, a waiting process should enter.
  3. Bounded Waiting: Finite wait before entering CS.

Race Condition

Race condition occurs when multiple processes access and modify shared data concurrently, and the outcome depends on the order of execution leading to inconsistent results.

Example:
Shared variable x = 5. Two processes run simultaneously:

P1: x = x + 1     P2: x = x - 1
Both read x=5.
P1 computes: 5+1=6   P2 computes: 5-1=4
P1 writes 6.  OR  P2 writes 4.
Final x = 6 or 4, but should be 5!

Race Condition Diagram:

             |
P2: READ(x)  COMPUTE(x-1)  WRITE(x=4)
Both read x=5 before either writes -- result is wrong!

Solutions: Mutex, Semaphore, Monitor, Peterson's Solution, Hardware atomic operations (Test-and-Set, Swap).

Conclusion: Critical sections must be protected using synchronization primitives to prevent race conditions and ensure data consistency in concurrent systems.

Q9. Explain Internal vs External Fragmentation in Memory Managem

Fragmentation is the wastage of memory space that reduces effective memory utilization. There are two types:

1. Internal Fragmentation

Occurs when allocated memory block is larger than requested, leaving unused space INSIDE the allocated partition.

+------------------+
| Process (7 KB)   |
|------------------|
| Wasted (3 KB)    |  <-- Internal Fragmentation
+------------------+
   Partition = 10 KB

Example: Partition=10KB, Process needs 7KB. Wasted = 10-7 = 3KB inside the partition.
Occurs in: Fixed partitioning, paging systems.

Occurs when total free memory is sufficient but exists in scattered non-contiguous blocks, so no single large block is available.

|Used|Free|Used|Free|Used|
|10K | 5K |8K  | 7K |6K  |
Total free = 12K but no contiguous 12K block!

Example: Free blocks of 5K, 7K exist (total=12K). Process needs 10K contiguous cannot allocate!
Occurs in: Variable partitioning, dynamic allocation.

Free spaceWasted inside partitionScattered outside

Solutions: Internal use smaller partition sizes or paging. External compaction (rearrange memory) or paging (fixed-size pages eliminate external fragmentation).

Conclusion: Internal fragmentation wastes space inside partitions; external fragmentation scatters free space. Paging is the most effective solution for external fragmentation.
